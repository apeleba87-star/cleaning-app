# 업체운영자 UX 개선 마스터 플랜

## 1. 문서 목적

이 문서는 `업체운영자` 관점에서 서비스의 속도와 사용성을 실제 판매 가능한 수준으로 끌어올리기 위한 실행 계획서다.

- 목표: 기능 추가보다 `업무 완료 속도`와 `오류 없는 흐름`을 우선 개선
- 범위: `business` 영역 핵심 화면 및 관련 API/데이터 로딩 구조
- 원칙: 측정 -> 개선 -> 검증 -> 배포 사이클 고정

---

## 2. 성공 기준 (KPI)

## 2.1 속도 KPI
- 첫 화면 체감 로딩 (LCP): 2.5초 이내
- 핵심 목록 화면 초기 데이터 표시: 1.5초 이내
- 필터/정렬 반응: 500ms 이내
- 버튼 액션 후 UI 반영: 300ms 이내 (낙관적 업데이트 우선)

## 2.2 사용성 KPI
- 핵심 업무 5개 평균 완료 시간 30% 단축
- "왜 안 되는지 모르겠다" 유형 오류/문의 50% 감소
- 모바일 업무 흐름 중 이탈률 30% 감소

## 2.3 비즈니스 KPI
- 데모 -> 유료 전환율 상승
- 2주 활성 유지율 상승
- 운영자 CS 건수 감소

---

## 3. 핵심 사용자 여정 정의

업체운영자 핵심 여정은 아래 3개에 집중한다.

1) 오늘 운영 확인
- 대시보드 진입
- 매장 상태 확인
- 요청/이슈/미완료 작업 처리

2) 인력 운영
- 사용자 등록/관리
- 매장 배정/근무여부/역할 정리

3) 성과 확인
- 평균 관리시간
- 미관리 매장
- 재무/인건비 상태

`핵심 여정 3개`를 먼저 완성하고, 이후 세부 기능 확장한다.

---

## 4. 현재 문제 요약 (진단)

## 4.1 속도
- 페이지마다 다건 조회 + 클라이언트 후처리 비중이 높음
- 일부 화면은 전체 새로고침 의존
- 동일 데이터 반복 조회로 체감 지연 발생

## 4.2 UX 일관성
- 확인/완료/반려/관리종료의 동작과 문구가 화면별로 다르게 보임
- 필터/기간/정렬 UI 패턴이 분산됨
- 오류 메시지가 해결 행동을 명확히 안내하지 못함

## 4.3 정보구조(IA)
- 기능은 많지만 업무 중심이 아닌 메뉴 구조
- "지금 무엇을 해야 하는지" 우선순위가 약함

---

## 5. 개선 전략 (3단계 로드맵)

## Phase 1 (1~2주): 체감 성능 + 즉시 피드백
목표: "느리다" 체감 제거

- 공통 원칙
  - 서버 페이징 우선
  - 필요한 컬럼만 select
  - 액션 성공 시 전체 reload 금지 (낙관적 업데이트)
  - 로딩/에러/빈 상태 UI 컴포넌트 통일

- 우선 화면
  - `business/dashboard`
  - `business/stores/status`
  - `business/users`

- 산출물
  - 느린 API 목록 및 개선안
  - 공통 스켈레톤/에러 컴포넌트
  - 화면별 응답시간 측정 로그

## Phase 2 (2~4주): IA 재정렬 + 화면 패턴 통일
목표: "어디서 뭘 해야 할지" 즉시 이해 가능

- 메뉴/화면 구조 재정의
  - 오늘 운영
  - 인력
  - 성과
  - 설정

- 화면 패턴 통일
  - 상단: 제목 + 핵심 KPI + 공통 필터
  - 본문: PC 테이블 / 모바일 카드
  - 하단: 페이지네이션

- 상태/액션 문구 표준화
  - 관리중/관리완료
  - 요청 확인/처리 완료/반려
  - 불가 상태는 이유 + 해결 행동 동시 노출

## Phase 3 (4~8주): 판매 가능 완성도
목표: 교육 없이 사용 가능한 운영 도구

- 온보딩 강화
  - 첫 로그인 3단계 가이드
  - 역할별 시작 체크리스트

- 리포트 강화
  - 평균 관리시간, 미관리 매장, 인력 가시성

- QA 체계화
  - 역할별 E2E 체크리스트
  - 배포 전 회귀검증 루틴

## 6. 화면별 개선 백로그 (업체운영자 중심)

## 6.1 대시보드 (`/business/dashboard`)
- 해야 할 일 우선 카드 고정 (긴급도 순)
- "오늘 미완료" 숫자를 클릭하면 해당 상세로 이동
- 데이터가 없을 때 다음 액션 버튼 제공

## 6.2 매장 상태 (`/business/stores/status`)
- 요약/상세 데이터 분리 로딩
- 모달 내 액션 후 부분 업데이트
- 필터(프렌차이즈/상태/기간) 반응속도 개선
- "완료 불가"는 사유를 버튼 근처에 선표시

## 6.3 사용자 관리 (`/business/users`)
- 검색 + 필터 + 정렬 + 페이징 공통 패턴 유지
- 편집/삭제/배정 액션 위치 고정
- 모바일 카드에서도 핵심 정보 1스크린 내 표시

## 6.4 평균 관리시간 (`/business/average-management-time`)
- 현재 구현 유지:
  - 전 매장 표시
  - 프렌차이즈 필터
  - 기간 선택
  - 5시간 초과 제외
- 추가 개선:
  - "데이터 기준 안내" 고정 문구
  - 조회 조건 유지(쿼리스트링/로컬 저장)

---

## 7. 데이터/성능 개선 체크리스트

- 인덱스 검토
  - attendance(store_id, clock_in_at, clock_out_at)
  - users(company_id, created_at)
  - stores(company_id, deleted_at)

- 쿼리 규칙
  - 와일드카드 select(*) 최소화
  - count 집계와 상세 조회 분리
  - 리스트는 페이지 단위 로드

- 캐시/재검증
  - 자주 바뀌지 않는 마스터 데이터는 재사용
  - 액션 후 필요한 섹션만 재조회

---

## 8. UX 문구/상태 표준

## 8.1 액션 라벨
- 요청 확인
- 처리 완료
- 반려
- 관리시작
- 관리종료

## 8.2 불가 상태 안내
- 금지 문구 + 이유 + 해결 행동
- 예시:
  - "금일 수행 미션이 남아 있습니다."
  - "체크리스트 2건, 요청란 1건 완료 후 관리종료 가능합니다."

## 8.3 Empty State
- "데이터 없음"만 출력 금지
- "다음 행동" 버튼 제공

---

## 9. 분석/측정 설계

- 이벤트 추적
  - 페이지 진입
  - 필터 변경
  - 핵심 액션 클릭/성공/실패
  - 액션 완료까지 소요시간

- 운영 로그
  - API 응답시간 p50/p95
  - 실패율
  - 사용자별 이탈 구간

---

## 10. 2주 실행 플랜 (즉시 실행용)

## Week 1
- 공통 성능 계측 삽입
- 대시보드/매장상태 느린 API 개선
- 공통 로딩/에러/빈 상태 컴포넌트 도입

## Week 2
- 사용자 관리/평균 관리시간 UI 패턴 통일
- 모바일 핵심 동선 클릭 수 축소
- 실사용자 테스트(3명) 후 보정 배포

---

## 11. 역할 및 의사결정 규칙

- PO/대표: KPI 우선순위 확정
- 개발: 성능/UX 구현
- 운영: 실제 업무 시나리오 제공
- 공통 원칙:
  - 새 기능 추가보다 기존 흐름 개선 우선
  - KPI 개선 없는 변경은 보류

---

## 12. 릴리즈 체크리스트

- [ ] 역할별 접근권한 정상 동작
- [ ] 모바일/PC 주요 화면 깨짐 없음
- [ ] 핵심 여정 3개 E2E 통과
- [ ] 느린 API p95 목표 달성
- [ ] 오류 문구가 해결 행동을 안내

이 문서는 "기능 나열"이 아니라 "업무 완료 경험"을 목표로 한다.
